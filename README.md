# Medical Record Eval Sandbox

An open-source web app where users load synthetic C-CDA patient records, write prompts, build golden sets, and run evals live against Claude.

**Stack**: Next.js 15 · TypeScript · pgvector (PostgreSQL 16) · Vercel AI SDK v4 · voyage-3.5 embeddings · Claude (generation + judge) · Upstash Redis (rate-limit + spend cap) · Vitest · Playwright

---

## Corpus: synthetic data only

All patient records are generated by [Synthea](https://github.com/synthetichealth/synthea) — no real patients, no PHI. Names carry numeric suffixes (`Agustin437 Hills818`), UUIDs are random, and dates are fictional. **Do not add real clinical records to this repository.**

## Local development

### Prerequisites

- Node 20+
- Docker (for pgvector)
- An Anthropic API key, a Voyage AI key, and an Upstash Redis database

### 1. Start the database

```bash
docker compose up -d
```

Starts PostgreSQL 16 with the pgvector extension on `localhost:5432`.

### 2. Configure environment

```bash
cp .env.example .env.local
```

Fill in `.env.local`:

```env
ANTHROPIC_API_KEY=sk-ant-...        # Claude generation + judge
VOYAGE_API_KEY=pa-...               # voyage-3.5 embeddings (billed from token 1)
DATABASE_URL=postgres://postgres:postgres@localhost:5432/medical_eval
UPSTASH_REDIS_REST_URL=https://...  # free tier at upstash.com
UPSTASH_REDIS_REST_TOKEN=...
SEED_URL=                           # see step 3 below
```

> **Voyage billing**: `voyage-3.5` is billed from token 1 — no free tier. Expect ~$0.001–0.01
> per RAG query at published rates.

> **Upstash**: The spend-cap and rate-limit guards require Upstash Redis even locally.
> Create a free database at [upstash.com](https://upstash.com) and copy the REST URL + token.

### 3. Seed the database (one-time)

The fastest path loads a pre-built SQL dump — embeddings are already included, so no Voyage
call is needed for seeding. Set `SEED_URL` in `.env.local` to the URL published on the
[shahroseaziz.com/builds](https://shahroseaziz.com/builds) entry for this project, then run:

```bash
npm run seed
```

**Alternative** — seed directly from the bundled Synthea fixtures (calls Voyage):

```bash
npx tsx scripts/ingest.ts --local src/lib/ccda/__fixtures__
```

The seed command is idempotent-safe: it rejects if the database already contains patient rows.
To re-seed, truncate first:

```sql
TRUNCATE patients, chunks, seed_meta CASCADE;
```

### 4. Install and run

```bash
npm ci
npm run dev
```

Open <http://localhost:3000>.

## Deployment (Vercel + Neon + Upstash)

### Database: Neon

1. Create a project at [neon.tech](https://neon.tech).
2. Copy the **pooled** connection string (add `?sslmode=require` if not already present).
3. Run the one-time seed from your local machine:

```bash
DATABASE_URL=<neon-connection-string> \
  SEED_URL=<seed-sql-gz-url> \
  npm run seed
```

`SEED_URL` is the gzip-compressed SQL dump URL published on the project's builds page (see above).

### Redis: Upstash

1. Create a Redis database at [upstash.com](https://upstash.com). The free tier is sufficient
   — the built-in spend cap limits live-run costs to $5/day and $1/hour server-side.
2. Copy the **REST URL** and **REST token**.

### Vercel

1. Import the repo into Vercel (or push to a branch that Vercel watches).
2. Set the following environment variables in the Vercel dashboard:

| Variable | Description |
|---|---|
| `ANTHROPIC_API_KEY` | Anthropic API key (Claude generation + judge) |
| `VOYAGE_API_KEY` | Voyage AI key (voyage-3.5 embeddings) |
| `DATABASE_URL` | Neon pooled connection string |
| `UPSTASH_REDIS_REST_URL` | Upstash Redis REST URL |
| `UPSTASH_REDIS_REST_TOKEN` | Upstash Redis REST token |

3. Deploy. No `vercel.json` is needed — Next.js 15 is auto-detected. The `/api/run` function
   timeout is set to 120 s via `export const maxDuration = 120` in the route file.

> **Plan note**: Vercel Hobby caps function duration at 60 s. Upgrade to Pro (or use a
> self-hosted Next.js server) to match the 120 s `maxDuration` configured for streaming runs.

### Fork caveat

GitHub Actions secrets (`ANTHROPIC` and `VOYAGE_AI`) are **not** copied to forks.
After forking, add your own secrets under *Settings → Secrets and variables → Actions*
before CI can run the fixture-seeding and eval-gate steps.

## Evals

Thresholds live in `evals/thresholds.yaml`. CI runs a full eval gate (`npm run eval:gate`) on
every push; the gate exits non-zero when any metric drops below threshold.

| Metric | Threshold |
|---|---|
| Faithfulness | ≥ 0.85 |
| Contains | 1.0 (exact match) |
| Judge kappa min | 0.40 (Landis-Koch "moderate") |

Golden sets, scorers, and results live in the corresponding subdirectories under `evals/`.

## Scripts

| Command | Description |
|---|---|
| `npm run dev` | Start development server |
| `npm run build` | Production build |
| `npm test` | Unit + integration tests (Vitest) |
| `npm run test:e2e` | End-to-end tests (Playwright) |
| `npm run seed` | Load pre-built SQL seed (needs `DATABASE_URL` + `SEED_URL`) |
| `npm run ingest` | Parse C-CDA XML, embed with Voyage, write to DB |
| `npm run eval:gate` | Run eval quality gate (live Claude + Voyage — never mocked) |
| `npm run generate:baseline` | Pin eval baseline (first run only) |
| `npm run compute:kappa` | Judge calibration kappa analysis |
| `npm run validate:evals` | Validate eval artifacts |
